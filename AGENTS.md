# 项目协作规则

## 文件删除安全规则

禁止 Codex 执行任何批量删除、递归删除或模糊匹配删除操作。

严禁使用以下命令或类似变体：

- `del /s`
- `rd /s`
- `rmdir /s`
- `Remove-Item -Recurse`
- `rm -rf`
- `rm -r`
- 带通配符的删除命令，例如 `Remove-Item *.log`、`del *.tmp`、`rm *.csv`
- 基于搜索结果、管道或循环的批量删除，例如 `Get-ChildItem ... | Remove-Item`、`find ... -delete`、`for ... Remove-Item`、`foreach ... Remove-Item`

如果确实需要删除文件，只能一次删除一个明确、完整、具体路径的文件。删除前应确认目标是单个文件，不是目录、通配符结果或批量匹配结果。

如果任务需要删除多个文件、整个目录、缓存目录、构建产物、日志目录或临时文件集合，Codex 必须停止操作，并向用户说明需要删除的内容，让用户手动确认或自行删除。

## Git 提交与推送规则

本项目远程仓库为 `https://github.com/tanyw666/Quantitative-Trading-System.git`。

Codex 不需要在每一个小阶段完成后都机械提交和推送，而应根据任务进展自行判断合适时机。

优先在以下情况执行提交和推送：

- 完成一个相对完整、可独立回滚的功能或修复。
- 用户明确要求保存、提交、推送或准备休眠/结束当前工作。
- 改动已经通过必要检查，适合形成一个稳定版本。
- 本轮工作耗时较长，继续前有必要创建恢复点。
- 用户可能切换设备、关闭电脑或需要 GitHub 作为备份。

如果只是非常小的临时改动、探索性调整、尚未验证的半成品，或者马上还会继续大幅修改，可以暂不推送，但应在回复中说明当前改动状态。

执行提交和推送时，应遵循以下流程：

1. 检查 `git status --short --branch`。
2. 只暂存与当前任务相关、应该进入版本库的改动。
3. 创建清晰的中文提交信息。
4. 推送到 GitHub 的 `main` 分支。
5. 在最终回复中说明提交哈希、主要改动和推送结果。

如果改动中可能包含隐私数据、账号密钥、大文件、行情缓存、临时结果或用户未明确希望公开的文件，推送前必须先提醒用户确认。
